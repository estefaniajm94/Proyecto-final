## Incidencia 1 — ViewModels acoplados a Context mediante getString()

### Problema
Varios ViewModels (`SettingsViewModel`, `HomeViewModel`) construían
mensajes de estado con literales de String hardcodeados en Kotlin.
Esto impedía testar los ViewModels en JVM puro sin Robolectric:
cualquier prueba que verificara `uiState.statusMessage` requería
un contexto Android real.

### Causa
Es habitual en proyectos Android acceder a recursos de strings
directamente desde Activity o Fragment, pero al mover esa lógica
a los ViewModels para seguir MVVM, se arrastró la dependencia de Context
sin identificarla como un problema.

### Solución aplicada
Se diseñó la interfaz `StringResolver` con un único método
`get(@StringRes resId: Int, vararg formatArgs: Any): String`.
La implementación real `AndroidStringResolver` envuelve `Context.getString()`.
En tests se usa `TestStringResolver`, que devuelve la clave del recurso
como string, sin necesidad de Context.
Se inyecta en todos los ViewModels afectados mediante constructor.
Los literales hardcodeados se movieron a `strings.xml`.

### Aprendizaje
Los ViewModels no deben importar `android.content.Context`.
Cualquier dependencia Android debe abstraerse detrás de una interfaz
inyectable para mantener la testabilidad en JVM.

### Evidencia
- `presentation/common/StringResolver.kt`
- `presentation/settings/SettingsViewModel.kt`
- `presentation/home/HomeViewModel.kt`
- `test/TestStringResolver.kt`
- `test/TrafficLightUiTest.kt`

## Incidencia 2 — UiState genérico sin uso real en el proyecto

### Problema
Existía una `sealed interface UiState<T>` genérica en `presentation.common`
que pretendía modelar los estados Loading / Success / Error de forma reutilizable.
En la práctica, ningún ViewModel la usaba: cada pantalla tenía su propio
data class de estado con campos específicos.

### Causa
Se creó anticipando un patrón de reutilización que nunca se materializó.
Cada pantalla requería estados distintos (por ejemplo, `AnalyzeFlowState`
con pasos intermedios de OCR) que no encajaban en el molde genérico.

### Solución aplicada
Se eliminó `UiState.kt` como código muerto.
Cada ViewModel mantiene su propio estado específico:
`AnalyzeUiState`, `HomeUiState`, `HistoryUiState`, etc.
Esto hace que cada estado sea más expresivo y tipado.

### Aprendizaje
No abstraer prematuramente. Un wrapper genérico de estado añade
indirección sin valor si los consumidores necesitan campos distintos.
Tres clases específicas son más claras que una genérica mal ajustada.

### Evidencia
- `UiState.kt` eliminado (visible en el diff del commit `16ca1cf`)
- `presentation/analyze/AnalyzeUiState.kt`
- `presentation/home/HomeUiState.kt`

## Incidencia 3 — Trazas de Log y medición de tiempos en la capa domain

### Problema
`AnalyzeAndPersistUseCase` contenía llamadas a `android.util.Log`
y mediciones de tiempo de ejecución (`System.currentTimeMillis()`).
Esto introducía una dependencia de Android framework en el dominio
y dejaba código de depuración en producción.

### Causa
Durante el desarrollo se añadieron trazas para medir el rendimiento
del motor de reglas. Se olvidó eliminarlas antes de considerar el código como final.

### Solución aplicada
Se eliminaron todos los `Log.d` y las mediciones de tiempo.
El dominio queda libre de imports `android.*`.
Los tests unitarios del motor de reglas cubren el comportamiento
sin necesidad de trazas de depuración.

### Aprendizaje
Los casos de uso en domain no deben importar nada de `android.*`.
Las mediciones de rendimiento deben hacerse en tests o con herramientas
de profiling, no con logs en producción.

### Evidencia
- `domain/usecase/AnalyzeAndPersistUseCase.kt`
- `test/AnalyzeViewModelOcrTest.kt` (adaptado tras la limpieza)

## Incidencia 4 — Literales hardcodeados en el BiometricPrompt

### Problema
Los textos del diálogo biométrico en `SettingsFragment` y `HistoryFragment`
estaban escritos directamente como `String` en el código Kotlin:
`"Acceso protegido"`, `"Usa tu huella o PIN para continuar"`, etc.
Esto violaba las buenas prácticas de Android (todos los textos visibles
al usuario deben estar en `strings.xml`) e impedía la localización.

### Causa
Los literales se escribieron rápidamente durante el desarrollo
para verificar que la biometría funcionaba, sin externalizarlos.

### Solución aplicada
Se añadieron las claves correspondientes en `strings.xml`
y se actualizaron ambos Fragments para usar `getString(R.string.*)`.

### Aprendizaje
Externalizar textos desde el primer momento es más eficiente
que hacer una pasada de limpieza al final. Android Studio marca
estos literales con warnings; conviene no ignorarlos.

### Evidencia
- `res/values/strings.xml` (claves biométricas añadidas)
- `presentation/settings/SettingsFragment.kt`
- `presentation/history/HistoryFragment.kt`

## Incidencia 5 — Home rediseñado: tarjetas de acción sin componente reutilizable

### Problema
El layout inicial de `fragment_home.xml` repetía el mismo bloque XML
cuatro veces (una por tarjeta de acceso rápido: Coach, Entrenamiento,
Historial, Recursos), con atributos duplicados y difícil de mantener.
Un cambio de estilo requería editar cuatro bloques.

### Causa
Las tarjetas se diseñaron inlineadas en el layout principal
sin extraer un componente o `include` reutilizable.

### Solución aplicada
Se creó `view_home_action_card.xml` como layout independiente
que define la estructura visual de cada tarjeta (icono, título, descripción).
Las cuatro tarjetas del Home usan este layout mediante `<include>`.
Se añadió `dimens.xml` para centralizar los valores de tamaño y espaciado
usados en el panel.
Se añadió también `bg_icon_container.xml` y los drawables de barra de severidad
para unificar el estilo visual de los semáforos.

### Aprendizaje
Extraer componentes XML reutilizables desde la segunda repetición,
no esperar a la cuarta. El costo de la extracción temprana es mínimo.

### Evidencia
- `res/layout/view_home_action_card.xml`
- `res/layout/fragment_home.xml` (simplificado con includes)
- `res/values/dimens.xml`
- `res/drawable/bg_icon_container.xml`
- `res/drawable/bg_severity_bar_*.xml`

## Incidencia 6 — EncryptedSharedPreferences: advertencia de API deprecated

### Problema
Al compilar aparecía un warning sobre `MasterKeys.getOrCreate()` marcado
como deprecated en favor de `MasterKey.Builder`.

### Causa
La API de `MasterKeys` fue deprecada en versiones recientes de la librería
`security-crypto`, aunque el comportamiento subyacente (AES256-GCM) es idéntico.

### Solución y decisión tomada
Se decidió conscientemente **no migrar** a `MasterKey.Builder` en este punto.
El cifrado es el mismo. La migración podría introducir problemas de compatibilidad
en dispositivos con versiones antiguas de Android sin aportar mejora de seguridad real.
La decisión queda documentada en `JUSTIFICACION_TECNICA.md` sección 6.

### Aprendizaje
No toda advertencia del compilador requiere acción inmediata.
Cuando el comportamiento es equivalente y el cambio tiene riesgo de regresión,
documentar la decisión de no migrar es tan válido como migrar.

### Evidencia
- `data/local/preferences/SecureSettingsDataSource.kt`
- `JUSTIFICACION_TECNICA.md` — sección 6 (Limitaciones aceptadas conscientemente)

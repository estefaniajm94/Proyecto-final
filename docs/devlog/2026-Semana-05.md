## Incidencia 1 - La suite unitaria dejo de reflejar la UI real

### Problema
`HistoryViewModelTest` fallaba en filtros y orden, dando la impresion de que el
Historial estaba roto.

### Causa
El problema no estaba en `HistoryViewModel`, sino en `TestStringResolver`:
seguia devolviendo un formato antiguo para `history_item_meta` mientras el
recurso real ya habia cambiado a `Origen: %1$s`.

### Solucion aplicada
Se alineo `TestStringResolver` con `strings.xml` para que el doble de test
represente la salida real de produccion.

### Aprendizaje
Los tests pueden fallar por deriva de dobles y fixtures, no solo por bugs del
codigo productivo. En una fase de cierre conviene validar primero el contrato
entre recursos, UI y tests.

### Evidencia
- `app/src/test/java/com/estef/antiphishingcoach/TestStringResolver.kt`
- `app/src/main/res/values/strings.xml`
- `HistoryViewModelTest`

## Incidencia 2 - Un test del motor heuristico seguia usando una calibracion antigua

### Problema
`RuleEngineTest` esperaba un score minimo de `16` para un caso de urgencia.

### Causa
La regla `URGENCY_THREAT` habia quedado calibrada en peso `14`, pero el test no
se habia actualizado con ese ajuste.

### Solucion aplicada
Se corrigio la expectativa del test para reflejar la configuracion vigente del
motor heuristico, manteniendo la comprobacion de que la senal `URGENCY_THREAT`
se detecta correctamente.

### Aprendizaje
No toda correccion de cierre implica tocar produccion; a veces el codigo esta
bien y lo incorrecto es la expectativa automatizada.

### Evidencia
- `app/src/test/java/com/estef/antiphishingcoach/RuleEngineTest.kt`
- `app/src/main/java/com/estef/antiphishingcoach/domain/heuristics/DefaultHeuristicRules.kt`

## Incidencia 3 - Historial recogia estado desde un punto de ciclo de vida incorrecto

### Problema
`lint` marcaba `RepeatOnLifecycleWrongUsage` en Historial.

### Causa
El collector del `uiState` arrancaba desde una ruta que partia de `onResume`
despues de autenticar, aunque la vista ya estuviera creada.

### Solucion aplicada
Se movieron lista, controles y `collectOnStarted` a `onBoundView`, dejando en
`onResume` solo la validacion de acceso biometrico.

### Aprendizaje
Cuando una pantalla mezcla proteccion local y coleccion reactiva, conviene
separar claramente la compuerta de acceso del arranque del render.

### Evidencia
- `app/src/main/java/com/estef/antiphishingcoach/presentation/history/HistoryFragment.kt`
- `app/src/main/java/com/estef/antiphishingcoach/presentation/common/UiExtensions.kt`
- `lintDebug`

## Incidencia 4 - El gate de lint seguia cayendo por deuda de UI y recursos

### Problema
`lintDebug` fallaba por varios errores:
- `android:tint` en vez de `app:tint`,
- llamada API 26 en `HomeFragment`,
- textos hardcodeados,
- composicion no internacionalizable,
- ausencia de reglas explicitas de backup.

### Causa
Era deuda acumulada de iteraciones anteriores: la app funcionaba, pero no habia
pasado aun una ultima fase de endurecimiento para entrega.

### Solucion aplicada
Se corrigieron y documentaron:
- migracion de `android:tint` a `app:tint`,
- uso de `DrawableCompat` para el icono avatar del menu,
- externalizacion de textos a `strings.xml`,
- mejora i18n en quiz, recursos y checklist,
- `backup_rules.xml` y `data_extraction_rules.xml`,
- limpieza de recursos muertos y pluralizacion de contadores del training.

### Aprendizaje
El cierre real de un proyecto Android no termina cuando el APK compila: termina
cuando compila, pasa tests y supera los gates de calidad del build.

### Evidencia
- `app/src/main/res/layout/*`
- `app/src/main/java/com/estef/antiphishingcoach/presentation/home/HomeFragment.kt`
- `app/src/main/AndroidManifest.xml`
- `app/src/main/res/xml/backup_rules.xml`
- `app/src/main/res/xml/data_extraction_rules.xml`

## Incidencia 5 - La validacion final dependia del entorno de ejecucion

### Problema
`installDebug` no pudo completarse durante el cierre tecnico.

### Causa
No habia ningun emulador o dispositivo conectado en esta sesion. El error
observado fue `No connected devices!`.

### Solucion aplicada
No se maquillo el resultado: se dejo documentado en `README`, `Plan_de_Pruebas`
y `Anexo_I` como bloqueo de entorno, no como fallo del proyecto.

### Aprendizaje
La documentacion final debe distinguir entre defectos de producto y limites del
entorno de validacion. Esa honestidad mejora la trazabilidad de la entrega.

### Evidencia
- `docs/Plan_de_Pruebas.md`
- `docs/Anexo_I_Manual_Instalacion.md`
- `README.md`
